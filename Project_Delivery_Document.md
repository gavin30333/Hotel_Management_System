## 一、 管理端概述 (Overview)

### 1.1 产品定位
本项目旨在构建一个连接酒店商户与终端消费者的信息交互桥梁，专注于酒店信息的高效管理与运营协作。管理端为商户提供便捷的资源维护工具，同时赋予管理员全面的审核与监管能力，确保平台信息的准确性与合规性。

### 1.2 核心角色
系统主要服务于以下两类用户：
- **商户 (Merchant)**: 负责酒店信息的全生命周期管理，包括录入、编辑、提交审核及日常维护。
- **管理员 (Admin)**: 负责全平台酒店信息的审核、发布管控及违规处理，拥有全局数据视野。
- **超级管理员 (Super Admin)**: 拥有所有管理员权限，并负责管理管理员账户（包括封禁、解封），保障系统管理团队的纯洁性。

### 1.3 技术架构
项目采用前后端分离的现代化架构：
- **前端**: React + UmiJS + Ant Design (Pro Components)，提供响应式且一致的用户体验。
- **服务端**: Node.js (Controller/Model 架构)，处理业务逻辑与数据持久化。
- **交互**: 基于 RESTful API 进行数据通信。

## 二、 核心功能模块 (Core Functional Modules)

### 2.1 用户认证与个人中心 (Authentication & User Center)
基于 `src/pages/Login` 和 `src/layouts` 实现：
- **登录/注册**: 提供统一的多角色入口，支持新商户入驻注册及管理员登录。
- **个人中心**: 支持用户修改用户名及查看当前角色标签。
- **安全设置**: 提供密码修改功能，保障账户安全。
- **系统设置**: 支持系统通知/邮件通知的开关控制及界面偏好设置。

### 2.2 酒店资源管理 (Hotel Resource Management)
基于 `src/pages/Hotel` 实现：
- **酒店列表**:
    - **多维度筛选**: 支持按状态（草稿/待审核/已上线/已下线）、星级、关键词进行精准搜索。
    - **数据展示**: 采用卡片式列表展示酒店基础信息、价格区间及状态标签。
- **信息维护 (CRUD)**:
    - **新增/编辑**: 提供分步式表单，支持酒店基础信息、房型信息、周边设施及入住政策的录入与修改。
    - **详情查看**: 采用抽屉式 (Drawer) 详情页，展示图文、房型及周边信息，操作流畅。
    - **删除**: 仅支持处于“草稿”及“已下线”状态的酒店进行删除操作，防止误删已上线数据。

### 2.3 管理员管理 (Administrator Management)
**仅限超级管理员 (Super Admin) 可用**，基于 `src/pages/AdminManagement` 实现：
- **管理员列表**: 查看所有注册的管理员账户及其当前状态（正常/已封禁）。
- **账户管控**:
    - **封禁**: 禁止特定管理员登录系统，收回其审核权限。
    - **解封**: 恢复被封禁管理员的登录及操作权限。
- **登录限制**: 系统在登录环节（`src/controllers/authController.ts`）强制校验用户状态，已封禁用户无法进入系统。

### 2.4 审核与发布体系 (Audit & Publishing System)
- **状态流转**: 实现了完整的生命周期管理：
    - `Draft` (草稿) -> `Pending` (待审核)
    - `Pending` -> `Online` (已上线) / `Rejected` (已驳回)
    - `Online` -> `Offline` (已下线)
- **审核操作 (Admin)**:
    - **审核通过**: 一键发布上线，信息即时对外可见。
    - **审核驳回**: 必须填写驳回原因，商户端可查看反馈并据此修改。
- **上下线管理**:
    - **提交审核**: 商户将草稿或整改后的信息提交审核。
    - **下线/重新上线**: 管理员可对已发布酒店进行即时下线处理；整改完成后可申请重新上线。

## 三、 权限管理体系 (Permission System)

### 3.1 角色隔离
- **菜单权限**: 基于角色动态渲染侧边栏菜单，例如普通商户无法看到全局管理入口。
- **数据权限**: 商户仅能查看和管理归属自己的酒店数据，管理员拥有全平台数据的管理权限。

### 3.2 操作粒度控制
基于 `HotelListPage` 及 `BasicLayout` 的逻辑实现按钮级权限控制：
- **Super Admin**: 拥有系统最高权限，除包含 Admin 所有权限外，独有进入“管理员管理”页面及操作管理员账户的权限。
- **Admin**: 拥有酒店管理权限，包括审核（通过/驳回）、强制下线、以及对任意酒店信息的编辑权限。
- **Merchant**: 仅拥有编辑（草稿/驳回态）、提交审核、删除（非上线态）权限。

## 四、 操作流程规范 (Operational Workflows)

### 4.1 酒店上线流程
1. **商户录入**: 商户填写酒店信息并保存为草稿。
2. **提交审核**: 商户确认信息无误后提交审核。
3. **管理员待办**: 管理员后台收到待审核任务。
4. **审核通过**: 管理员审核通过，系统自动发布上线。

### 4.2 异常处理流程
1. **审核驳回**: 管理员发现信息问题，填写原因并驳回。
2. **查看原因**: 商户查看驳回原因。
3. **修改信息**: 商户根据反馈修改信息。
4. **再次提交**: 商户再次提交审核。

### 4.3 下线召回流程
1. **违规/维护**: 发现酒店违规或需维护。
2. **执行下线**: 管理员或商户执行下线操作。
3. **隐蔽处理**: 酒店在 C 端不可见。
4. **整改重上**: 整改完成后申请重新上线。

## 五、 界面与交互设计 (UI/UX)

### 5.1 视觉规范
全站采用 Ant Design 统一设计语言，遵循标准化的色彩体系与布局规范，确保视觉一致性。

### 5.2 交互反馈
- **全局状态**: 关键操作提供全局 Loading 状态。
- **即时提示**: 操作成功或失败均有 Toast 提示。
- **防误触**: 危险操作（如删除、下线）均配置二次确认弹窗 (Popconfirm)。

### 5.3 响应式布局
页面适配不同尺寸屏幕，列表与详情展示在各类设备上均能保持良好的阅读体验。
